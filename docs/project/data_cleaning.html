Author

Overview

This document performs all data cleaning and preprocessing steps on the Lightcast 2024 job postings dataset. After running all cells, the cleaned df object is ready to use — copy this file and continue writing your own analysis directly below Section 5.

Dataset Context

The Lightcast 2024 dataset is pre-filtered to data, analytics, and tech occupations (~72,000 postings, 131 columns). Key quirks to be aware of:

Issue Detail
TITLE Hash/ID string — not human-readable. Use TITLE_RAW for job titles.
COMPANY Numeric ID — not human-readable. Use COMPANY_NAME or COMPANY_RAW.
SALARY / SALARY_FROM / SALARY_TO ~55–58 % missing by design (employers often omit salary). Do not drop these columns.
MAX_EDULEVELS_NAME 77 % missing — use MIN_EDULEVELS_NAME instead.
LIGHTCAST_SECTORS_NAME 75 % missing — use NAICS_2022_6_NAME for industry.

3. Data Cleaning

Starting shape: (72498, 131)

3.1 Drop Tracking & Redundant Columns

We remove metadata columns (hash IDs, URLs, internal tracking) and older NAICS / SOC hierarchy levels that are superseded by the 2022 / 2021 versions we keep.

Columns retained:

  • NAICS_2022_2 through NAICS_2022_6 (and their _NAME variants) — current industry classification
  • SOC_2021_2 through SOC_2021_5 (and _NAME variants) — current occupation classification
  • All LOT_* occupation taxonomy columns
Dropped 18 columns: ['ID', 'URL', 'ACTIVE_URLS', 'ACTIVE_SOURCES_INFO', 'DUPLICATES', 'LAST_UPDATED_TIMESTAMP', 'LAST_UPDATED_DATE', 'NAICS2', 'NAICS3', 'NAICS4', 'NAICS5', 'NAICS6', 'SOC_2', 'SOC_2_NAME', 'SOC_3', 'SOC_3_NAME', 'SOC_5', 'SOC_5_NAME']
Shape after step 3.1: (72498, 113)

3.2 Drop High-Missing Columns (with Salary Protection)

We drop columns where more than 50 % of values are missing, but explicitly protect salary and education columns from this rule.

Why protect salary columns? SALARY, SALARY_FROM, and SALARY_TO are each ~55–58 % missing because many employers choose not to post salary information — this is normal for job postings data, not a data quality issue. The remaining ~42 % (≈ 30,000 rows) provides a sufficient sample for salary analysis. Dropping them would silently break all downstream salary work.

Protected columns: ['SALARY', 'SALARY_FROM', 'SALARY_TO', 'ORIGINAL_PAY_PERIOD', 'MIN_YEARS_EXPERIENCE', 'MAX_YEARS_EXPERIENCE', 'LIGHTCAST_SECTORS', 'LIGHTCAST_SECTORS_NAME', 'MIN_EDULEVELS', 'MIN_EDULEVELS_NAME', 'MAX_EDULEVELS', 'MAX_EDULEVELS_NAME']
Dropped 0 high-missing non-protected columns.
Shape after step 3.2: (72498, 113)

3.3 Remove Duplicate Postings

We deduplicate on the combination of job title + company + state + posting date. This removes re-posted or syndicated listings that represent the same position.

Deduplication key: ['TITLE_RAW', 'COMPANY_NAME', 'STATE_NAME', 'POSTED']
Removed 4,926 duplicate rows (6.8% of pre-dedup total).
Shape after step 3.3: (67572, 113)

3.4 Build Unified Salary Column (SALARY_MID)

We create a single salary column by taking the midpoint of SALARY_FROM and SALARY_TO. Where the midpoint is unavailable we fall back to the SALARY column.

Important: Missing salary values are left as NaN — we do not impute them with the median or mean. All downstream salary analyses should use .dropna(subset=["SALARY_MID"]) or filter to df[df["SALARY_MID"].notna()] to work only with real data, preserving the true salary distribution.

Created SALARY_MID = (SALARY_FROM + SALARY_TO) / 2  [fallback: SALARY]

Salary coverage:  30,037 postings with data (44.5%)
Plausible range:  $10,617 – $470,000
Median:           $111,800
Mean:             $114,245
